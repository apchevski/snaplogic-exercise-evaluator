# Changelog

## [Unreleased]

- **Cognito now supports optional TOTP (authenticator-app) MFA, so admins/mentors can add a second factor.** The user pool (`cognito-auth` module) was `mfa_configuration = "OFF"`, which is why enabling an authenticator on a user failed with `User does not have delivery config set to turn on SOFTWARE_TOKEN_MFA` — software-token MFA has to be enabled on the pool before any user can register or set it. The pool is now `OPTIONAL` with `software_token_mfa_configuration { enabled = true }`; the hosted UI drives self-enrollment (associate → verify → enabled), so no app code changed. Needs `terraform apply`; existing users keep signing in without MFA until they enroll. Flip to `mfa_configuration = "ON"` to require it for everyone.

- **Dashboard student search now matches on project name too, and the box says so.** The search box (which previously filtered students by name or project space) now also matches the SnapLogic **project** name, so you can find a student by any of student name, project space, or project. Its placeholder changed from "Search by student or project space" to "Search by student, project space, or project". Students whose project is unset (named after the student) were already findable by name.

- **Every action button that kicks off a grading or prep run now asks for confirmation first, using the same `ConfirmModal` as Delete/Archive/Remove.** Previously **Prep All Exercises** and the per-row **Prep** (Exercises page), and **Grade all exercises**, **Regrade**, and the per-exercise **Grade** (student detail page) fired immediately on click — an accidental click started a background job (and, for grading, spent API credits). Each now opens a styled confirmation dialog (header, X, Cancel, and a blue primary confirm button) spelling out what the run does before it starts; confirming closes the dialog and the job runs in the background with its usual status pill. The Dashboard **Grade** button is unchanged (it already opens the grade-scope picker), and read-only/reversible actions (file downloads, Unarchive) stay one-click.

- **Archive confirmation is now an in-app dialog matching the Delete dialog, replacing the native browser `confirm()`.** Clicking **Archive** on an exercise opened the browser's built-in prompt (`d35z1iswzmrtga.cloudfront.net says…`); it now opens the same styled `ConfirmModal` used by Delete, with a header, an X, a Cancel button, and a blue **Archive** confirm button (primary, not red — archiving is reversible). The `ConfirmDeleteModal` component was generalized to `ConfirmModal` (new optional `confirmClassName` and `busyLabel` props, defaulting to the old delete styling) and reused by the exercise-delete and student-remove dialogs. Unarchive stays a one-click inline action (no confirmation).

- **Modals now scroll inside the dialog, fixing a close-on-scrollbar bug on small screens.** On a laptop-height screen the Add/Edit Exercise dialog was taller than the viewport and the scrollbar belonged to the backdrop; clicking that scrollbar registered as a click outside the dialog and closed it. The dialog is now capped to the viewport (`max-height`) with the body as the scroll region, so the scrollbar lives inside the dialog and clicking it no longer dismisses it. Applies to every modal (Add/Edit Exercise, Add Student, delete confirmations, grade scope) since they share the `.modal` layout; the short ones are unaffected because their content still fits.

- **Add/Edit Exercise dialog: friendlier, UI-first fields — no more folder slugs or `# Header` boilerplate.** The dialog stopped mirroring the on-disk file names now that S3 (not the git `exercises/` folders) is the source of truth:
  - **New "Exercise Name" field** replaces writing a literal `# Task 07 – Router Basics` H1 inside the description. It's the human-readable pipeline name; the dialog stores it as `description.md`'s H1 for you, so prep's pipeline lookup is unchanged.
  - **The "Folder name" (slug) field is gone from create.** The stable exercise id is derived from the name automatically (`slugify`) — nobody types `task_07_router_basics` anymore. The slug is still the permanent id and is still immutable after create.
  - **Fields renamed to plain language:** `description.md` → **Description** (just the prompt body now), `notes.md` → **AI Guidance**. On edit, the existing H1 is split back out of `description.md` into Exercise Name so the two fields round-trip. Backend contracts (`description_md` with an H1, `slug`) are untouched. Matching the Add Student dialog treatment: **description.md**, **Folder name**, and (per task type) **Output filenames**, **Triggered Task name**, and **Request scenarios** now carry a `req-star` `*`; **notes.md** and **Input files** drop their "(optional)"/"(required)" text and show just the field name.

- **Student detail: never-graded students now show their project path instead of "graded never".** The meta line under the student name used to read `graded never` when there was no report; it now shows the SnapLogic project path alone (no date), e.g. `Interworks-Partner/Training_Program_Demo/Matt Murdock`. The path is computed server-side in `GET /v1/students/{slug}` (new `student_project_path` on the student meta = `org/space/project`, mirroring what grade.py records) so it's available before the first grade — the frontend can't build it because `/v1/config` 403s the student role. Once graded, the line is unchanged (`path · graded <date>`).

- **Dashboard: Project is now its own column, and the Add Student dialog marks required fields.** The student's SnapLogic project name used to appear as a subline under their name (only when it differed) — it now has a dedicated, sortable **Project** column between **Project Space** and **Total Points**, showing the effective project (the student name when unset). In the **Add Student** dialog: the **Student email** field is restyled to the same width as the others (its `input[type="email"]` was never covered by the field CSS) and drops the "(optional)" suffix; the two mandatory fields — **Student name** and **Project space** — get a red `*` marker, and Project space is now enforced (the Add Student button stays disabled until both are filled). Email and Project remain optional (Project still defaults to the student name).

- **New `SOLUTION_OVERVIEW.md`: a one-read map of the whole solution.** Covers the architecture (with a Mermaid diagram GitHub renders), the two-layer evaluation model, data model and S3 layout, auth/roles, the key flows (grade, prep, add student, authoring, deletes), CI/CD, local dev, and a list of invariants a new contributor — human or AI — must not break. Linked from the README's *Architecture & design notes* and the project-layout tree. A visual counterpart (architecture diagram page) was published as a Claude Artifact.

- **Read-only `student` role: students can sign in and watch their grades.** A third Cognito group joins `admin`/`mentor`:
  - **Student logins are created from the Add Student dialog.** The dialog gains an optional **Student email** field. When set, `POST /v1/students` (new optional `email` key) invites the student into the pool's new `student` group via `AdminCreateUser` — Cognito emails them a temporary password, the hosted UI forces a change on first sign-in, and the API never sees a password. The email is validated, lowercased, stored on the student card (`email`), and the whole registration fails as a unit if the login can't be created (card rolled back; 409 when a login for that email already exists, 400 on a malformed address, 503 when `USER_POOL_ID` isn't configured). Registration without an email is unchanged.
  - **Students see everything mentors see, minus every action.** The role matrix in `backend/src/api.py` grants `student` the read routes the dashboards need (student list/detail/report history, exercise list, input-file downloads) and 403s everything else — including `GET /v1/config`, job polling, and `GET /v1/exercises/{slug}`, which carries `notes.md` (instructor hints). The UI hides what the backend forbids: no Add Student / Grade / Regrade / Remove buttons, no edit pencils, no prep warnings (`useCanGrade()` in `auth.ts`; the backend matrix is the real gate).
  - **Deleting a student also deletes their login.** `DELETE /v1/students/{slug}` removes the Cognito user recorded on the card's `email` (best-effort, reported as `deleted.login`), so an orphaned login can't keep reading grades after an admin purge. The worker's card refresh carries `email` forward on regrades so that link survives.
  - Infra: new `student` Cognito group (`cognito-auth`), API Lambda IAM gains `cognito-idp:AdminCreateUser/AdminAddUserToGroup/AdminDeleteUser` scoped to the pool, and the API Lambda gets `USER_POOL_ID`. Tests need moto's `cognitoidp` extra (`requirements.txt`). Needs `terraform apply` + backend + frontend redeploy.

- **Header brand: "Exercise Evaluator" now matches the "SnapLogic" wordmark.** Same font size, weight, and color (was smaller, muted text); the divider between the two stays.

- **Dashboard polish: cleaner button labels, a dedicated Not Graded column, and a quieter Last Graded fallback.** The **Add Student**, **Grade**, **Remove**, and (Exercises page) **Delete** buttons drop their trailing "…". The "N not graded" chip inside the Total Points cell is gone; never-graded exercise counts now get their own sortable **Not Graded** column between Missing and Last Graded (same hover explanation as the old chip). Students who have never been graded show "—" in Last Graded instead of "never graded".

- **Permanently remove students and exercises from the UI (admin only, red buttons, confirmation dialogs).** Complements Archive (which stays the reversible option) with true deletion that leaves no data behind in AWS:
  - **Remove a student.** A red **Remove…** button on each dashboard row (admins only; mentors never see it) opens a confirmation dialog spelling out the blast radius. Confirming calls the new `DELETE /v1/students/{slug}`, which deletes the student card and all REPORT history rows from DynamoDB, their grade-job rows and lock, and every object under `students/<slug>/` in S3 — **all versions**, since the bucket's versioning would otherwise keep recoverable copies for 90 days. 409 while a grading for that student is queued/running.
  - **Delete an exercise.** A red **Delete…** button next to Archive (admins only) confirms, then calls the new `DELETE /v1/exercises/{slug}`: purges `exercises/<slug>/` and `exercise-resources/<slug>/` from S3 (all versions), deletes the EXERCISE row, prep-job rows and lock, and **scrubs the exercise's result out of every student's live report** — task entry removed from `report.json`, its section dropped from `report.md`, counts/points recomputed with the same rules grading uses, and the denormalized student card refreshed. Older report versions are left alone (they're the students' grading history). 409 while a prep is queued/running.
  - **Tombstones stop image resurrection.** An exercise whose folder still ships in the container image can't just have its row deleted — the image copy would reappear on the next listing and get re-seeded into S3 by the next prep. Deleting such an exercise keeps a minimal tombstone row (`slug` + `deleted` flag + who/when); the listing, single-exercise GET/PUT, resource downloads, prep and grading all treat tombstoned slugs as gone, and the worker prunes them from its working tree exactly like archived ones (`_prune_archived_exercises` → `_prune_excluded_exercises`). `POST /v1/exercises` with a tombstoned slug re-creates the exercise (the fresh row replaces the tombstone). Exercises that live only in S3 (the normal UI-authored case) are deleted without any tombstone.
  - What a delete can't reach (documented in the code): CloudWatch log lines (age out with the log group's retention) and the nightly `exercises-backup/` git snapshot — the deleted exercise drops out of the snapshot on the workflow's next run, but git history retains prior snapshots.
  - Infra: two new `DELETE` routes in `api-gateway/locals.tf`, API CORS gains `DELETE`, and the API IAM policy gains `s3:DeleteObject`/`s3:DeleteObjectVersion` on the three entity prefixes plus prefix-scoped `s3:ListBucketVersions` (version purge needs to enumerate versions; this makes `students/*` enumerable by the API role, previously deliberately unlistable). Needs `terraform apply` + backend + frontend redeploy.

- **Per-student project space and project name, set when adding the student.** The dashboard's Add Student form became a dialog (**Add Student…**) with three fields: student name, **project space** (prefilled with the configured default via the new `GET /v1/config`, which returns only non-secret settings), and an optional **project** name for when the SnapLogic project isn't named exactly after the student. `POST /v1/students` stores both on the student card (`space` resolved to the actual value at registration, so the dashboard's Project Space column is no longer empty) and verifies the project exists at that exact location before registering. From then on the card dictates every grading run: `POST /v1/gradings` resolves `body space override → card → env default` into the job payload, the worker passes both into the evaluator, and the card refresh writes them back (which also backfills the space for students registered before this change on their next grading). The evaluator/CLI grew a matching `--project` flag (`python -m evaluator run`, `evaluator.grade plan`); manifest and `report.json` now record the `project` alongside `project_space`. `SNAPLOGIC_STUDENT_PROJECT_SPACE` is therefore now just the default for the dialog/legacy cards. Students whose project differs from their name show it as a subline under their name on the dashboard. New route in `api-gateway/locals.tf` (`GET /v1/config`) — needs `terraform apply` + backend + frontend redeploy.

- **The dashboard toolbar button is now register-only ("Add Student"), and registration verifies the SnapLogic project exists.** Typing a name and submitting used to open the grade-scope picker (with a "Register only" footer button for new students) — redundant, since every row already has a **Grade…** button that opens the same picker. The form now just calls `POST /v1/students` and the picker's register path is gone (`GradeScopeModal` lost its `isNew`/`onRegister` props). Behind the scenes, `POST /v1/students` now checks SnapLogic before creating the card: one GET-only asset-list call against `<org>/<student project space>/<student>` — a 404 comes back as a clear `400 "No project named '<name>' exists in the '<space>' project space"` instead of registering a card every grading run would fail on; SnapLogic being unreachable is a 502, and environments with no SnapLogic credentials configured (local dev, tests) skip the check. The secret-loading helper moved from `worker.py` to `common.py` (`load_secrets_into_env`) so both Lambdas share it. Infra: the API Lambda now gets `SECRET_ARN` + `secretsmanager:GetSecretValue` (it never had the SnapLogic creds before) — needs `terraform apply` + backend + frontend redeploy.

- **Silence the VSCode GitHub Actions extension's "Context access might be invalid" warnings in the deploy workflows.** `AWS_DEPLOY_ROLE_ARN` / `AWS_REGION` are loaded at runtime from `.github/deploy.vars` via `$GITHUB_ENV`, which the extension can't see statically. The jobs that reference `${{ env.… }}` now declare empty job-level placeholders so the linter knows the names exist; runtime values from `$GITHUB_ENV` override job-level env, so behavior is unchanged. Placed at job level (not workflow level) so the empty `AWS_REGION` never reaches the backend test job's boto3/moto region resolution.

- **Grade-scope picker: "all / none" links replaced with a select-all checkbox.** The exercise list in the Grade dialog now has a master checkbox at the top of the list (SnapLogic Monitor style): check to select every exercise, uncheck to clear, with an indeterminate dash when only some are selected.

- **Pick which exercises a grading covers, register students without grading, and surface never-graded exercises everywhere.** Three connected changes to how students enter the system:
  - **Grade-scope picker.** The dashboard's per-row **Grade** button and the "grade a new student" form now open a scope dialog (all active exercises preselected): keep everything for a full run, or check a subset — including for a brand-new student, who previously could only start with an all-exercises run. `POST /v1/gradings` accepts a new `tasks` array (validated, deduped, archived slugs rejected; mutually exclusive with `task`; a one-element array collapses to the existing single-`task` semantics), and the worker runs the scoped slugs sequentially inside one job/lock, merging each result into the student's stored report and summing usage/cost onto the JOB item (REPORT history rows record the multi-slug scope in `tasks_scope`). A full run still refreshes the AI Overall summary; scoped runs leave it untouched.
  - **Register without grading.** The scope dialog's **Register only** button calls the new `POST /v1/students` (mentor or admin), which creates the student card with zero grades and $0 spent (409 if the student already exists). Grading later fills the card in; the registration stamp (`registered_by`/`registered_at`) and any registered `space` survive the card refresh.
  - **"Not graded" is now visible.** Registered exercises a student has never been graded on — registered-only students, or exercises added to the set after their last run — show as grey **not graded** cards on the student detail page (each with its own **Grade** button, plus a **Grade all exercises** action when nothing is graded yet), a count badge in the Grade Summary panel, and an "N not graded" chip on the dashboard row. Stored totals are not rewritten: the points denominator still reflects what the last grading run covered, and grows as the missing exercises get graded.
  - Infra: new `POST /v1/students` route in `api-gateway/locals.tf` — needs `terraform apply` + backend + frontend redeploy.

- **Fix report-text edits always failing with `Internal Server Error` (while still persisting after a refresh).** `PATCH /v1/students/{slug}/report` rewrites `report.json`/`report.md` in S3 and then stamps the edit onto the student card with `dynamodb:UpdateItem` — but the API Lambda's IAM policy never granted `UpdateItem` (only `GetItem`/`PutItem`/`DeleteItem`/`Query`), so every edit saved the S3 report and then 500'd on the DynamoDB stamp. That's why the change "came back" after a refresh even though the UI reported an error, and why the denormalized `overall_summary` on the dashboard card could go stale. Moto-based tests don't enforce IAM, which is how it slipped through. The policy now includes `dynamodb:UpdateItem` — needs `terraform apply`. Frontend fix in the same change: a failed summary save now shows its error inline under the editor's Save/Cancel buttons instead of only in the page-top banner, which is scrolled out of view when editing a task card further down the page (that made the task-edit failure look like a silent hang on "Saving…").

- **Edit AI-written report text from the student detail page (mentor or admin).** Every task card gets a pencil button next to **Regrade** (and the Overall summary gets one too) that opens an inline textarea to rewrite the AI's summary/explanation — for when the wording is off but the verdict is right, at $0 instead of a paid re-grade. Saving calls the new `PATCH /v1/students/{slug}/report` (`{overall_summary}` or `{task, summary}`), which rewrites the latest stored `report.json` in place, keeps the `report.md` Overall paragraph in sync, refreshes the denormalized `overall_summary` on the student card, and stamps `…_edited_by`/`…_edited_at` provenance on the edited text plus `report_edited_by/at` on the student META item. Verdicts, points, and deductions are untouched, and a later regrade of a task intentionally replaces its edited summary with fresh AI text (edits to *other* tasks and the Overall survive the merge). A student graded single-task-only (no Overall yet) shows a "No overall summary yet." placeholder with the pencil, so a mentor can write one by hand instead of paying for a full run. Infra: new `PATCH /v1/students/{slug}/report` route, `PATCH` added to API CORS, and an API IAM `s3:PutObject` statement scoped to `students/*/report.json` + `students/*/report.md` only (everything else under `students/` stays worker-only) — needs `terraform apply` + backend + frontend redeploy.

- **Exercises are now managed end-to-end in the web UI; S3 is the source of truth.** Follow-up to (and partial supersession of) the "Add new exercises from the web UI" entry below — exercise content no longer has git as its long-term home:
  - **Create/edit/archive from the UI (admin only).** The Add dialog became a full create/edit dialog (an **Edit** button on every row loads the authored content via the new `GET /v1/exercises/{slug}`; saves go through the new `PUT /v1/exercises/{slug}` — partial update of description/notes/config, plus adding input files via presigned PUTs and deleting existing ones). **Archive/Unarchive** buttons soft-delete: the worker prunes archived exercises from its working tree after materialize, so they're skipped by prep, excluded from grading and from the points denominator — but nothing is ever deleted from S3, and the API rejects prep/single-task-grade requests for archived slugs (400). Archived rows render greyed-out with a badge.
  - **`task.json` is no longer hand-written.** The dialog has a task-type selector — *file writer single output* ("auto": prep detects the lone writer, unchanged), *file writer multi/custom* (output filenames + `exact`/`columns_only` match mode), or *triggered task* (task name, defaulted to `<pipeline> Task`, plus scenario rows of name + `param=value` pairs). The validated config is stored on the EXERCISE row (`task_config`); the worker synthesizes `task.json` from it (plus the env-derived `solution_pipeline_path`) into the merged tree before every prep sync, and the existing reconciler takes it from there. This closes the old gap where UI-created triggered-task exercises stalled at `needs_task_json_triggered` with no cloud path forward.
  - **Git demoted to a create-only seed.** Prep jobs now additively migrate authored files (`description.md`, `notes.md`, `resources/*`) from the image into S3 (`S3Store.seed_authored_files` — never overwrites an existing S3 key, never deletes), and the Exercises listing flips precedence: the S3 copy of authored content wins over the image copy, so UI edits show even while the image still ships the original. Existing exercises migrate by clicking **Prep All** once after deploying. The prep survey now preserves API-authored row attributes (`task_config`, `archived`, `created_by`, …) instead of wiping them on every put.
  - **Durability, production-style.** Data bucket: versioning enabled (noncurrent versions kept 90 days) + `prevent_destroy`; DynamoDB table: `prevent_destroy` (PITR + deletion protection were already on). New nightly `backup-exercises.yml` workflow exports `s3://…/exercises/` into `exercises-backup/` in this repo (one-way snapshot, commits only on change, restore runbook in the file header) using the existing GitHub OIDC role — no new credentials.
  - Infra: `GET`/`PUT /v1/exercises/{slug}` routes, API CORS gains `PUT`, API IAM gains `s3:DeleteObject` scoped to `exercises/*/resources/*` only (generated artifacts stay worker-only), `DATA_BUCKET` added to `deploy.vars`. Needs `terraform apply` + backend + frontend redeploy, then one **Prep All** run to migrate.

- **Add new exercises from the web UI (admin only).** An **Add New Exercise** button next to *Prep All Exercises* opens a dialog mirroring the manual authoring flow: `description.md` text (required, H1 = pipeline name; the folder name is auto-suggested from it), optional `notes.md` text, and optional student input files. Saving calls the new admin-only `POST /v1/exercises`, which writes the markdown to S3 under `exercises/<slug>/` and returns presigned PUT URLs so the browser uploads the input files straight to S3 (same 6 MB-Lambda-ceiling reasoning as resource downloads, in reverse). Because `S3Store.materialize_exercises` already overlays the whole S3 `exercises/` prefix onto the image tree before every job, prep and grading pick the new exercise up with **zero worker changes** — it behaves exactly like a committed folder. `GET /v1/exercises` now also lists S3-authored exercises (a slug counts as S3-authored when S3 holds its `description.md` — prep-generated artifacts never include one), resource downloads presign the S3 copy directly, and the `task`/`slug` validation on `POST /v1/gradings` / `POST /v1/preps` accepts them. A folder later committed with only `task.json` (e.g. a triggered-task exercise authored in the UI) still shows its S3 description and input files. Infra: new route, an API IAM `s3:PutObject` statement scoped to the three authored filenames only (`description.md`, `notes.md`, `resources/*` — generated artifacts stay worker-only), `s3:ListBucket` extended to the `exercises/*` prefix, and a CORS configuration on the data bucket for the presigned PUTs. Needs `terraform apply` + backend + frontend redeploy. Limitation documented in the README: the UI takes no `task.json`, so multi-output and triggered-task exercises still get theirs via git.

- **Drop the Slug column (Exercises page) and the Needs Prep column/badge (student grades + detail pages).** The slug added no information over the exercise title, and "needs prep" is an operator problem, not a grade — the verdict surfaces are now strictly Pass / Fail / Missing. Unprepped exercises are surfaced as warnings instead: the student grades table shows a ⚠ next to the affected student's points chip (hover for the count and the fix), and the student detail page shows an amber warning banner naming the skipped exercises and pointing at the Exercises page's Prep action. Task cards with status `needs_prep` no longer render on the detail page (they carried no verdict); they're listed in the banner instead. The search box on Exercises still matches slugs, since untitled exercises display their slug as the title.

- **Regrade a single exercise from the student detail page.** Every task card on `/students/<slug>` now has a **Regrade** button that queues a grading job for just that exercise (`POST /v1/gradings {student, task}` — the API accepted `task` since the cloud pivot, but no UI ever sent it). One Claude call instead of one per exercise, so a re-run is proportionally cheaper and faster; the job status pill and the refreshed report appear in place. The API now validates the `task` slug against the exercise folders (400 on unknown, mirroring `/v1/preps`), and the shared per-student grade lock still applies (409 while any grade job for that student is in flight). **Also fixes a latent data-loss bug in cloud single-task grading:** the report renderer merges a single-task result into the existing `grades/<student>/report.{md,json}`, but the worker Lambda's `/tmp` starts empty, so the "merge" would have produced a report containing only the regraded task and clobbered every other result on the student's card. The worker now downloads the previous report version from S3 first (new `S3Store.materialize_report`, keys read off the student's DynamoDB META item, legacy `report_json` attribute included; `LocalStore` no-op). New tests cover the store download, the worker's single-task flow (with and without a previous report), and the API validation/pass-through.

- **Fix `GET /v1/exercises/{slug}/resources/{filename}` returning 500 on the first download of each file.** The lazy S3 mirror probes with `HeadObject` before uploading, but the API Lambda role had no `s3:ListBucket` — and without it S3 masks "key doesn't exist" as **403 Forbidden** instead of 404, so the probe's error escaped the missing-object handling and the request died with `{"message":"Internal Server Error"}`. Once a file had been mirrored once, later downloads worked, which is why only first downloads failed. Fixed in two layers: `api-gateway` IAM gains a `s3:ListBucket` statement condition-scoped to the `exercise-resources/*` prefix (HeadObject now gets a real 404; the API still can't enumerate worker-owned prefixes), and `_sync_resource_to_s3` also treats a 403 probe as "can't see it, upload anyway" so future IAM drift degrades to one redundant re-upload instead of a 500. Regression test simulates the 403-masked HeadObject. Needs `terraform apply` + backend redeploy.

- **Fix the student detail page never showing per-exercise results.** Clicking a student's name opens `/students/<slug>`, which is supposed to render the full report — per-task verdicts, AI summaries, deductions with reasoning, failing gates, bonus answers — but it always fell back to "No stored report yet" with only the totals and the overall comment. Root cause: a key-name mismatch — `store.upload_report()` returned the S3 keys labeled `report_md`/`report_json`, the worker spread those onto the student's DynamoDB META item, but `GET /v1/students/<slug>` looked up `report_json_key` (the name the API tests and frontend types always used), so the report S3 fetch never ran. The store labels are now `report_md_key`/`report_json_key`, and the API additionally falls back to the legacy `report_json` attribute so students graded before the fix show their reports without regrading. New regression tests cover both attribute names.

- **Downloadable exercise input files on the Exercises page.** Exercises that hand the student input data (e.g. `Task1.zip`, `Leads.csv`) now show those files in a new **Files** column — one click fetches a 5-minute presigned S3 URL from the new `GET /v1/exercises/{slug}/resources/{filename}` route and the browser downloads straight from S3 (streaming a ~4 MB zip through the Lambda would flirt with the 6 MB response ceiling once base64-encoded). The route lazily mirrors the image's copy into S3 under `exercise-resources/<slug>/` (MD5-compared via ETag, so repeat downloads cost one HeadObject; the prefix is deliberately outside the worker-owned `exercises/` prefix that `S3Store.materialize` re-downloads wholesale, and is the only prefix the API role can PutObject to). `GET /v1/exercises` now returns a `resources` array (`filename` + `size_bytes`) per exercise via `evaluator/tasks.py: list_exercise_resources()`; path traversal is rejected by `exercise_resource_path()` (404). Infra: new route in `api-gateway/locals.tf` + a scoped `s3:PutObject` statement.
- **New convention: student-facing input files live in `exercises/<slug>/resources/`** (`.claude/conventions/exercise-resources-folder.md`).
