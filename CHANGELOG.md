- **Stop Terraform from perpetually re-diffing the Lambda `image_uri`.** Both `aws_lambda_function` resources (`api-gateway`, `sqs-worker`) now have `lifecycle { ignore_changes = [image_uri] }`. Terraform declares the image at the `:latest` tag, but `deploy-backend.yml` owns code deploys and repoints each live function to an immutable commit-SHA tag via `update-function-code` — so every `terraform plan` reported a spurious `…:<sha> → …:latest` in-place update (a benign no-op redeploy). Ignoring `image_uri` lets the two pipelines stop fighting and keeps infra plans clean; Terraform still creates the functions at `:latest` on first apply.
- **Fold `ci.yml` into the deploy workflows so each domain tests itself on PRs, then deploys on main.** The standalone `ci` workflow (which ran python/frontend/terraform checks in parallel, and — critically — did *not* gate the deploys, so a deploy could ship on `main` while `ci` was red) is deleted. Each deploy workflow is now a self-contained pipeline of separate jobs — gate jobs run on `pull_request`, and the deploy job `needs:` them and is `main`-only: `deploy-backend` = `test` (ruff + pytest, $0) → `build` (image → ECR) → `deploy` (repoint both Lambdas), the last two main-only since the "build" pushes to ECR rather than compiling locally; `deploy-frontend` = `test` (vitest) → `build` (tsc type-check + vite bundle, uploads `dist` as an artifact, no AWS access) → `deploy` (downloads the artifact, syncs S3 + invalidates CloudFront); `deploy-infra` = `validate` (terraform `fmt -check -recursive infra` + offline `validate` on both the production and bootstrap roots) → `plan` → `apply`. Net effect: a red gate can no longer ship, and each workflow reads as an explicit test→build→deploy staging. Trade-offs: checks are now path-scoped (a frontend-only PR won't run backend tests), and feature-branch pushes without an open PR aren't tested (open the PR to get CI) — if you later add branch protection, require each workflow's gate job individually rather than a single `ci` check.

- **Output matching is now column-order-insensitive** for every exercise. The `output_match` hard gate previously compared the header as an ordered list, so a student who emitted the right columns in a different order (e.g. `Department ID, Employee ID, Name` instead of `Employee ID, Name, Department ID`) failed even when the data was correct. Now the gate compares the **column-name set** (same names, same count, any order); when the order differs it **realigns the student's rows to the solution's column order by name** before the row-multiset comparison, so a pure reordering passes. Only a missing column, an extra column, a different column count, or genuinely different row data fails. `columns_only` mode likewise compares the column-name set in any order. Row order was already ignored; now column order is too. (Implemented in `evaluator/hard_gates.py` via `_columns_match` / `_build_column_index_map` / `_reorder_row`; rules text, README, and architecture notes updated.)
